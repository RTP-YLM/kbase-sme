### ข้อ 1 — วัตถุประสงค์และขอบเขต

ผู้ให้บริการประมวลผลข้อมูลในนามของลูกค้าเพื่อ:
- จัดเก็บและค้นหาเนื้อหาจากเอกสารองค์กรที่ลูกค้าอัปโหลด
- ให้บริการระบบถามตอบด้วย AI สำหรับพนักงานของลูกค้า
- บันทึกประวัติการใช้งานเพื่อการปรับปรุงคุณภาพ

### ข้อ 2 — ประเภทข้อมูลที่ประมวลผล

| ประเภท | ตัวอย่าง | วัตถุประสงค์ |
|---|---|---|
| เนื้อหาเอกสาร | นโยบาย HR, คู่มือ, SOP | สร้าง knowledge base |
| ข้อมูลผู้ใช้ | อีเมล, รหัสพนักงาน | ระบุตัวตนเพื่อ access control |
| ประวัติการสอบถาม | คำถาม, คำตอบ | ปรับปรุงคุณภาพ, รายงานรายเดือน |
| ข้อมูลการใช้งาน | จำนวนครั้ง, latency | billing, SLA monitoring |

**ไม่ประมวลผล:** ข้อมูลที่อ่อนไหว (สุขภาพ, ศาสนา, ชีวมิติ) เว้นแต่ระบุไว้ในสัญญาแยก

### ข้อ 3 — มาตรการความปลอดภัย

ผู้ให้บริการดำเนินการดังต่อไปนี้:
- [ ] เข้ารหัสข้อมูลระหว่างส่ง (TLS 1.2+)
- [ ] เข้ารหัสข้อมูลในฐานข้อมูล (AES-256 via Supabase)
- [ ] ควบคุมการเข้าถึงด้วย JWT + Row-Level Security
- [ ] สำรองข้อมูลรายวัน เก็บ 30 วัน
- [ ] ไม่ใช้ข้อมูลลูกค้าในการเทรน AI model

### ข้อ 4 — การจัดเก็บและระยะเวลา

- **ที่จัดเก็บ:** Supabase (Singapore, AWS ap-southeast-1)
- **ระยะเวลา:** ตลอดสัญญา + 90 วันหลังสิ้นสุด
- **การลบ:** ลูกค้าร้องขอได้ตลอดเวลา คืนข้อมูล JSON ภายใน 7 วัน ลบถาวรภายใน 30 วัน

### ข้อ 5 — ผู้ประมวลผลรายย่อย (Sub-processors)

| บริการ | บริษัท | วัตถุประสงค์ | ที่ตั้ง |
|---|---|---|---|
| Vector Database | Supabase Inc. | จัดเก็บ embedding + เนื้อหา | Singapore |
| LLM (default) | Alibaba Cloud | สร้างคำตอบ | International |
| Redis Cache | Self-hosted | Semantic cache | Server ลูกค้า |

*ลูกค้าสามารถขอ on-premise option เพื่อให้ข้อมูลอยู่ในไทยได้ (Enterprise package)*

### ข้อ 6 — สิทธิ์ของเจ้าของข้อมูล

ลูกค้ามีหน้าที่แจ้งสิทธิ์ต่อพนักงาน ผู้ให้บริการสนับสนุน:
- สิทธิ์เข้าถึงข้อมูล (Right of Access)
- สิทธิ์แก้ไข (Right to Rectification)
- สิทธิ์ลบ (Right to Erasure) — ดำเนินการภายใน 30 วันนับจากร้องขอ

## ส่วนที่ 2: PDPA Checklist

### สำหรับลูกค้า (ก่อน go-live)

**ด้านกฎหมาย:**
- [ ] มีนโยบาย Privacy Policy สำหรับพนักงาน
- [ ] แจ้งพนักงานว่ามีการบันทึกประวัติการสอบถาม
- [ ] กำหนดผู้รับผิดชอบข้อมูล (Data Controller) ภายในองค์กร
- [ ] ถ้าเอกสารมีข้อมูลส่วนบุคคล (เช่น ชื่อพนักงาน) — ระบุฐานทางกฎหมายที่ใช้ประมวลผล

**ด้านเทคนิค:**
- [ ] ตรวจสอบว่าเอกสารที่อัปโหลดไม่มีข้อมูลส่วนบุคคลที่ไม่จำเป็น
- [ ] กำหนด access_level ของแต่ละเอกสารให้เหมาะสม
- [ ] แจ้งพนักงานที่มีสิทธิ์ใช้ระบบ

**ด้านการดำเนินงาน:**
- [ ] กำหนดว่าใครเป็น admin ของระบบ
- [ ] ทบทวนประวัติ query log รายเดือน
- [ ] มีขั้นตอนรับมือกรณีข้อมูลรั่วไหล (Data Breach) ภายใน 72 ชม.

### สำหรับผู้ให้บริการ (ตรวจรายไตรมาส)

- [ ] อัปเดต security patch ของ dependencies
- [ ] ทดสอบ backup restore
- [ ] ทบทวน sub-processor list
- [ ] ตรวจสอบ access log ผิดปกติ
- [ ] ต่ออายุ TLS certificate
