# Feasibility, Pricing, and Technical Concerns Analysis
> **SME Internal Knowledge Base RAG Platform**
> วิเคราะห์ความเป็นไปได้เชิงธุรกิจ โครงสร้างราคา สถาปัตยกรรมโมเดล และข้อกังวลด้านเทคนิคสำหรับการนำเสนอโซลูชัน RAG แก่ SME ไทย

---

## 1. การประเมินความเป็นไปได้เชิงธุรกิจ (Business Feasibility)

การแข่งขันในตลาดระบบค้นหาข้อมูลภายในองค์กรด้วย AI (RAG) สามารถวิเคราะห์ได้โดยแบ่งกลุ่มคู่แข่งและเปรียบเทียบจุดยืนของโครงการนี้ดังนี้:

### 1.1 การเปรียบเทียบกับคู่แข่งในตลาด (Competitive Landscape)
*   **Global DIY SaaS (เช่น Chatbase, MeetCody):** ราคาถูกมาก (1,000 - 1,400 บาท/เดือน) แต่เป็นระบบบริการตนเอง (Self-Service) ที่ต้องการการตั้งค่าเองทั้งหมด และไม่มีบริการเชื่อมต่อเฉพาะแบบ On-Premise หรือการปรับปรุงเอกสารให้
*   **Thai E-commerce Chatbots (เช่น Zwiz.ai):** เก่งเรื่องระบบตอบแชทปิดการขาย ระบบบิล บรอดแคสต์ แต่ไม่ได้เน้นการสืบค้นข้อมูลลึกจากคลังความรู้ของบริษัท (SOP/Policy)
*   **Software Houses / System Integrators (SI):** รับทำระบบ RAG เฉพาะทางแต่ราคาสูงมาก (500,000 - 1,500,000+ บาท) ซึ่งเกินเอื้อมของ SME ขนาดกลางและเล็ก

> [!NOTE]
> **จุดยืนที่เป็นไปได้มากที่สุด (Strategic Positioning):** 
> มุ่งเน้นบริการแบบ **"Done-For-You (DFY)"** หรือบริการติดตั้งพร้อมใช้ โดยการใช้ความเชี่ยวชาญภาษาไทยและการเชื่อมต่อกับ **LINE OA** เป็นจุดดึงดูดหลักสำหรับ SME ไทยที่ไม่มีทีมเทคโนโลยีของตัวเอง

## 3. การจัดสรรรูปแบบ LLM (Local LLM vs Cloud API Provider)

การเลือกเทคโนโลยีของ LLM ตามโครงสร้างราคาจะส่งผลต่อความคุ้มทุนและอัตรากำไร (Margin) ดังนี้:

### 3.1 ตารางการใช้โมเดลตามระดับราคา
1.  **Starter (4,900 THB/เดือน):** **ใช้ Cloud API (Pay-as-you-go) เท่านั้น**
    *   *โมเดลแนะนำ:* `Qwen3.5-Flash`, `GPT-4o-mini`
    *   *เหตุผล:* ปริมาณการใช้งานไม่สูง ค่า API ต่อเดือนเฉลี่ยไม่เกิน 200 - 500 บาท การรันบอทบน VPS ราคาประหยัดจะช่วยให้บริษัททำกำไรได้สูง (Margin > 85%)
2.  **Growth (14,900 THB/เดือน):** **ใช้ Cloud API**
    *   *โมเดลแนะนำ:* `GPT-4o-mini`, `Claude 3.5 Haiku`
    *   *เหตุผล:* แม้พนักงาน 50 คนจะใช้งานเยอะขึ้น แต่ค่า API คาดว่าจะไม่เกิน 1,500 - 2,500 บาท/เดือน ซึ่งยังคุ้มค่ากว่าการลงทุนซื้อหรือเช่า GPU Server
3.  **Business (39,900 THB/เดือน):** **เหมาะสำหรับ Local LLM (On-Premise) หรือ Dedicated Private Cloud**
    *   *โมเดลแนะนำ:* `Qwen2.5-7B-Instruct`, `Llama-3-8B-Instruct`
    *   *เหตุผล:*
        *   **On-Premise:** ลูกค้าจัดซื้อฮาร์ดแวร์ GPU (มูลค่าประมาณ 60,000 - 100,000 บาท) ไว้ที่ออฟฟิศตัวเองผ่าน Setup Fee เรานำโมเดลไปรันในเครื่องเซิร์ฟเวอร์ภายในของลูกค้า ซึ่ง**เราจะไม่มีต้นทุนค่า LLM API รายเดือนอีกเลย** รายเดือนจึงเป็นค่าบำรุงรักษาและบริการหลังการขาย (SLA)
        *   **Private Cloud:** เช่าเครื่อง GPU คลาวด์แยกเฉพาะ (ค่าใช้จ่ายประมาณ 10,000 บาท/เดือน) เพื่อรันโมเดลแยกให้ลูกค้าแต่ละราย ทำให้ระบบมีความปลอดภัยและเก็บรักษาความลับข้อมูลได้ดี

## 4. ข้อกังวลและอุปสรรคสำคัญ (Key Concerns & Technical Risks)

> [!WARNING]
> ประเด็นด้านล่างนี้คือความท้าทายหลักที่ทีมพัฒนาและฝ่ายขายต้องระวังในการส่งมอบระบบให้แก่ SME จริง

### 4.1 ความยากในการเตรียมข้อมูลนำเข้า (Data Ingestion & Cleaning)
*   **การตัดคำภาษาไทย (Thai Word Segmentation):** 
    *   ในโค้ดปัจจุบัน ([document_loader.py:L132](file:///D:/Project%20Dear/kbase-sme/src/document_loader.py#L132)) ใช้ฟังก์ชัน `words = text.split()` ซึ่งมีปัญหาอย่างมากกับภาษาไทยที่ไม่มีช่องว่างระหว่างคำ ทำให้การแบ่ง Chunk คลาดเคลื่อนและตัดคำแบบสุ่ม
    *   *แนวทางแก้ไข:* ต้อง refactor ไปใช้ไลบรารีตัดคำไทย เช่น `PyThaiNLP` หรือใช้กลยุทธ์แบ่งประโยคแทนการนับคำอังกฤษ
*   **เอกสารที่ไม่มีคุณภาพ (Scanned PDFs / Images):**
    *   นโยบายบริษัทในอดีตมักเป็นไฟล์รูปภาพแสกนเก็บไว้ใน LINE หรือโฟลเดอร์ ซึ่งตัวอ่าน PDF ทั่วไปดึงข้อความไม่ได้ จำเป็นต้องใช้ระบบ OCR ที่แม่นยำภาษาไทยระดับสูง
*   **ตารางข้อมูล (Structured Tables):**
    *   ข้อมูลตารางเบิกจ่าย ตารางค่าตอบแทน หรือขั้นตอนในตาราง Excel มักจะเสียโครงสร้างเมื่อถูกแปลงเป็นข้อความดิบ ทำให้ AI ดึงบริบทแถว/คอลัมน์สับสนและตอบคำถามคลาดเคลื่อน

### 4.2 ข้อกังวลการจัดเก็บเวอร์ชันและการอัปเดตข้อมูล (Document Lifecycle & CRUD)
*   **ปัญหาข้อมูลเก่าทับซ้อน (Vector Duplicate Pollution):**
    *   หากพนักงานอัปเดตนโยบายเวอร์ชันใหม่และรันระบบนำเข้าข้อมูลโดยไม่มีการลบ Chunk ข้อมูลนโยบายเก่าใน Supabase ข้อมูลเก่าและใหม่จะปนกัน ทำให้ AI มีโอกาสตอบคำถามผิดพลาดสูง
    *   *แนวทางแก้ไข:* จำเป็นต้องรีบทำระบบ **Source Catalog** เพื่อจับคู่ว่าแต่ละไฟล์ประกอบด้วย Chunk ID ใดบ้าง เมื่อมีการแก้ไขหรือลบไฟล์ ระบบต้องสามารถเคลียร์ Chunk เดิมออกจาก Vector DB ได้แบบอัตโนมัติ

### 4.3 ข้อกังวลด้านการยอมรับและความพร้อมขององค์กร (User Adoption & Data Quality)
*   **Garbage In, Garbage Out:** หากคู่มือปฏิบัติงานขององค์กรเองยังมีความขัดแย้งกัน หรือเขียนไม่เคลียร์ AI ก็จะตอบแบบไม่ชัดเจนตามไปด้วย จำเป็นต้องมีการตรวจสอบคุณภาพเอกสาร (Document Audit) ร่วมกับลูกค้าในขั้นตอน Setup
*   **ความคุ้นเคยกับเครื่องมือ:** พนักงานอาจไม่ยอมเข้ามาใช้งานหน้าจอแชทส่วนตัวแยกเพิ่ม การเชื่อมโยงบริการผ่าน **LINE OA** จึงเป็นจุดสำคัญในการเพิ่มสถิติการใช้งานจริง (Adoption Rate)
